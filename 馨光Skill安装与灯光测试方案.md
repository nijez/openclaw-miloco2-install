# 馨光 Skill 安装与灯光测试方案

本文是第二阶段本地草稿。第一阶段安装流程、安装脚本和页面 4 步主流程保持不变。

## 第二阶段目标

第二阶段只做一件事：让小龙虾通过馨光 Skill 控制馨光设备，实现自然语言驱动的 AI 设计灯光效果。

当前主流程简化为：

1. 安装馨光 Skill。
2. 启动馨光本地灯控服务。
3. 让用户选择米家家庭。
4. 小龙虾读取该家庭下的设备列表。
5. 用户直接用自然语言控制灯光。

第二阶段不并入第一阶段默认安装流程，也不加入首页正式入口，除非后续另行确认。

## 必须保留：米家家庭选择

多家庭选择是第二阶段必须保留的安全步骤。

流程：

1. 米家账号绑定成功后，读取米家家庭列表。
2. 如果只有一个家庭，直接继续。
3. 如果有多个家庭，必须让用户选择。
4. 不允许默认使用第一个家庭。
5. 用户选择家庭后，再读取该家庭下的设备列表。
6. 后续灯光控制只在这个家庭范围内进行。

如果当前工具无法切换或确认家庭，不要假装成功，应提示联系技术人员处理。

## 用户主流程

用户不需要每次选择产品类型、设备 DID 或具体设备。

推荐表达：

```text
完成前面步骤后，可以继续安装馨光 Skill。
安装完成后，请先选择要使用的米家家庭。
之后你可以直接告诉小龙虾想要的灯光效果，例如“客厅设计一个马尔代夫灯光效果”。
小龙虾会读取该家庭下的馨光设备，并通过馨光 Skill 执行灯光效果。
```

用户可以直接输入：

```text
客厅设计一个马尔代夫灯光效果。
二楼客厅来一个森林晨光。
卧室做一个适合睡前放松的灯光。
全屋灯带做一个朋友聚会氛围。
保存当前灯光效果到快照 3。
```

## 内部匹配逻辑

设备匹配逻辑保留在内部，不作为用户主流程展示。

内部可以保留：

1. 优先匹配当前米家家庭。
2. 根据用户语句识别房间。
3. 根据设备列表找到 RGBCW 设备。
4. 如果多个候选设备无法判断，再询问用户。
5. 如果找不到设备，再提示检查米家账号、家庭、设备在线状态。

不要在用户页面或普通安装指令里写成：

```text
请选择产品类型
请选择馨光RGBCW幻彩灯带
请选择具体 did
```

这些内容只保留在测试和排查文档中。

## 单次测试规则

真实灯光测试仍保留单次测试规则：

1. 一次只发送一个灯光请求。
2. 请求发出后等待用户现场确认。
3. 不自动重试。
4. 不自动关灯。
5. 不自动切换颜色继续验证。
6. 不用 `miloco-cli` 直接反向控制 RGBCW 设备。
7. 不控制其它设备。

如果 API 返回 `success:false`，但用户确认现场灯光已经变化，应记录为：

```text
PHYSICAL_SUCCESS_API_FALSE
```

如果出现连续变色、关灯再亮或多条命令叠加，应记录为：

```text
UNSTABLE_MULTIPLE_COMMANDS
```

## 暂停扩展的复杂逻辑

以下问题记录为研发修复项，不作为当前用户主流程继续扩展：

1. 自动修复 token。
2. 自动 patch 二进制。
3. 自动猜设备。
4. 自动诊断设备失败。
5. API false 后连续补发命令。
6. 把设备识别规则写到用户页面。
7. 要求用户每次选择产品类型或设备 DID。

## 给小龙虾的安装指令

正式指令见：

```text
docs/发给小龙虾的馨光Skill安装指令.txt
```

执行目标：

1. 不重复执行第一阶段安装流程。
2. 不修改第一阶段安装脚本。
3. 下载固定版本的第二阶段安装脚本。
4. 安装馨光 Skill。
5. 启动馨光本地灯控服务。
6. 检查米家家庭列表。
7. 如果需要，让用户选择米家家庭。
8. 读取所选家庭下的设备列表。
9. 安装完成后，提示用户直接用自然语言描述灯光需求。

## 验收标准

第二阶段通过验收，需要满足：

1. 馨光 Skill 已安装。
2. 馨光本地灯控服务已启动。
3. 多家庭场景不会自动使用第一个家庭。
4. 用户选择家庭后，小龙虾能读取该家庭下的设备列表。
5. 用户可以用自然语言提出灯光需求。
6. 小龙虾能在当前家庭范围内调用馨光 Skill。
7. 馨光设备有实际灯光变化。
